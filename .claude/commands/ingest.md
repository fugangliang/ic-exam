# /ingest — 撮影画像の取り込みと問題JSON化

## 実行前提（必読）

- この手順は **Sonnet等の高速モデルでの実行を想定して設計されている**。
  品質は手順と機械チェックで担保する。以下を厳守すること:
  1. **手順を上から順に実行する。スキップ・順序変更・「効率化」のための省略を禁止する**
  2. **判断に迷ったら考え込まずに `[要確認]` を付けて先へ進む**（判断はRFの仕事）
  3. 各ステップの検証コマンドは**実際に実行し、出力を確認してから**次へ進む
- 問題文・解説の創作・言い換え・「改善」を禁止する。画像にある文言の転記と再構成のみ。

## 手順

### 1. 画像の収集と重複除外

以下の場所から新しいスキャンファイル（jpg/jpeg/png/heic/pdf）を集める:

- `IC_scan/`（リポジトリ直下。**これが標準経路** — RFがAirDrop受信後ここに置く。iCloud Driveは不使用と確定済み）
- `~/Downloads/`（AirDrop直置きの補助経路）。ただし**ファイル名が「スキャン」「Scan」「adf」「IC」で始まる、または過去24時間以内に追加された画像/PDFのみを候補とし、移動前に必ず候補一覧をユーザーに提示して確認を取る**（無関係なダウンロードの誤取り込み防止）

`inbox/` に直接置かれたファイルはそのまま処理対象。

**移動前に必ずファイル重複チェックを実行する:**

```
python3 scripts/check_ingest_dupes.py --check IC_scan/*
```

- `[DUP]` と出たファイルは**取込済み**（内容ハッシュ一致。ファイル名が違っても検出される）。
  `inbox/` に移動せず**その場で削除**し、報告時に「重複スキップ」一覧として明示する
- `[NEW]` のみを `inbox/` に移動して処理する

### 2. ID採番（手で数えない）

出典略号と分野略号を決めたら、**必ず次のコマンドで開始番号を取得する**:

```
python3 scripts/next_ids.py <出典略号>-<分野略号>
```

- 分野略号: 歴史=his / 計画=pla / エレメント=ele / 構造・構法=str / 環境・設備=env / 法規=law / 表現=exp / 販売・コンサル=sal /
  色彩と造形=iro / 家具=kag / ウィンドウトリートメント=wnd
  （2026-07-14 過去問題集の実処理で追加。新カテゴリが出た場合はこの一覧に追記すること）
- 出力された「次のID」から連番で振る。既存IDの目視カウント・記憶に基づく採番を禁止する

### 3. 構造化（1ファイルずつ）

各画像/PDFを読み、問題と解説を以下のスキーマでJSON化する:

```json
{
  "id": "<出典略号>-<分野略号>-<連番3桁>",
  "category": "9分野のいずれか",
  "subcategory": "小分類",
  "source": "出典（例: 問題集2026 p.45 / 2022年度過去問）",
  "body": "問題文",
  "choices": ["肢1", "肢2", "肢3", "肢4"],
  "answer": 0,
  "explanation": "正答理由＋各誤答肢がなぜ誤りかを肢ごとに記述",
  "has_figure": false,
  "figure_ref": null,
  "difficulty": 2
}
```

- answer は0始まりのindex
- explanation は解説ページの内容の**再構成**（一次ソース=問題集解説）。正答理由に加え、各誤答肢がなぜ誤りかを肢ごとに構造化する
- 図版が不可欠な問題は `has_figure: true` とし、bodyは取り込むが P1では出題対象外になる（アプリ側で自動除外）

**空欄分解形式（過去問題集の穴埋め問題。RF確認・決定 2026-07-13）**:
この問題集は「1問につき空欄ア〜エの4箇所、各3択」形式。**空欄ごとに1問として分解する**
（詳細ルールは `CLAUDE.md` §問題JSONスキーマ の「空欄分解形式」を必ず参照）。
1画像（1問カード）から最大4件のJSONを生成する。次の順で作る:

1. 解答・解説ページから4空欄すべての正答語を確認する
2. 分解対象の空欄を1つ選び、body原文をコピーして**その空欄だけ `【　】` を残し、
   他の空欄は正答語で埋めて地の文にする**
3. choices はその空欄の3択を原文のまま使う。answer は正答の0始まりindex
4. explanation は解答・解説ページの当該空欄の説明文（＋キーワード欄の定義で補ってよい）
5. これを4空欄（ア/イ/ウ/エ）ぶん繰り返す。id 末尾は a/i/u/e、連番は4件で共通
6. category/subcategory はページ見出しから、difficulty は「重要度」の★の数から機械的に決める（迷いは生じない）

**1問JSON化するごとに、次の転記チェックを機械的に実施する**（暗算・記憶に頼らず画像と突合）:

| # | チェック | 不合格時の処置 |
|---|---|---|
| C1 | 問題文・選択肢の文字がすべて判読できたか | 判読不能箇所を `[要確認: 判読不能]` として本文に残す |
| C2 | 数値・単位・固有名詞を画像と1文字ずつ突合したか | 確信が持てない値に `[要確認]` を付ける |
| C3 | answer の index は「解説ページが正答と明言している肢」と一致するか | 解説に正答の明言がない/矛盾する場合 `[要確認: 正答不明]` |
| C4 | explanation は解説ページに書いてある内容だけで構成したか | 解説にない理由づけを書いていたら削除 |
| C5 | 全4肢が揃っているか。図表参照問題は has_figure: true にしたか | 肢が欠けていたら `[要確認: 肢欠落]` |

### 4. 品質ルール（CLAUDE.mdの最重要ルールに従う）

- **推測補完の絶対禁止**。文字が読み取れない・解答が判別できない・解説が欠けている場合は、その問題の `explanation` 冒頭に `[要確認: 理由]` を付け、処理終了時の報告で「人間レビュー必要」一覧として明示する
- 数値・固有名詞は画像の原文最優先。読取確信度が低い数値は `[要確認]` を付ける
- 1問でも判読不能なら飛ばさず `[要確認]` で残す（欠番はID連番のズレとして後で発覚しにくいため）
- **迷ったら `[要確認]`**。過剰に付ける分には害がない（RFがレビューで外す）。付け損ないだけが事故になる

### 5. 検証と保存

1. バッチ単位で `data/questions/<出典略号>_<日付YYMMDD>.json` に保存（形式: `{"questions": [...]}`）
2. `python3 scripts/validate_questions.py data/questions/*.json` を実行し、スキーマエラー・ID重複・**問題文重複**（正規化したbody＋choicesの一致＝同一問題の二重取込）がゼロであることを確認（既存ファイル横断でチェックされる）
3. 問題文重複が出た場合は新規側のJSONから該当問題を削除して再検証する（既存側を残す）
4. `python3 scripts/check_ingest_dupes.py --register inbox/*` で処理済み画像を台帳に登録する
   （**台帳登録を削除より先に必ず行う**。ハッシュが台帳に残るため、画像削除後も再送重複は検出できる）
5. 画像の後始末（ディスク節約のため原則削除。RF決定 2026-07-13）:
   - 検証を通過し `[要確認]` なしで取り込めた画像 → **削除する**（processed/ に残さない）
   - `[要確認]` 問題を含む画像 → `processed/` に移動して残す（原本照合ができなくなるため。
     人間レビューで解消したら削除してよい）

### 6. 最終チェックリストと報告

報告の前に、以下をすべて満たしていることを確認する（1つでも未実施なら該当手順に戻る）:

- [ ] `check_ingest_dupes.py --check` を実行した（手順1）
- [ ] `next_ids.py` でID開始番号を取得した（手順2）
- [ ] `validate_questions.py` が「エラー 0 件」で終了した（手順5-2）
- [ ] `check_ingest_dupes.py --register` を実行した（手順5-4）
- [ ] inbox/ と IC_scan/ が空になっている（削除 or processed/ 退避済み）

処理結果を表で報告する: ファイル数 / 重複スキップ数 / 抽出問題数 / 分野別内訳 / `[要確認]` 件数と該当ID一覧。

最後に必ず案内する:
> 生成したJSONをiPhoneにAirDropし、アプリの⚙（データ管理）→「問題データimport」で取り込んでください。

## 禁止事項

- `data/` `inbox/` `processed/` `IC_scan/` を git add / push しない（.gitignore済みだが明示的にも禁止）
- 問題文・解説の創作・言い換えによる「改善」（原文の再構成のみ）
- 検証コマンドの実行を省略して「実行した」と報告すること
